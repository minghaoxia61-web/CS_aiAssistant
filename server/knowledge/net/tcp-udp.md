# TCP 与 UDP

## TCP 与 UDP 对比

TCP（传输控制协议）和 UDP（用户数据报协议）都是传输层协议，但特性截然不同。

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠（确认、重传） | 不可靠 |
| 顺序 | 有序到达 | 无序 |
| 速度 | 较慢 | 快 |
| 头部开销 | 20 字节 | 8 字节 |
| 流控/拥塞控制 | 有 | 无 |
| 通信方式 | 一对一 | 一对一/多/广播 |
| 应用 | HTTP, FTP, SMTP | DNS, 视频, 游戏, QUIC |

## TCP 报文段结构

TCP 头部共 20 字节（不含选项），主要字段：
- 源端口、目的端口（各 2 字节）
- 序号 seq（4 字节）：标识数据流中字节位置
- 确认号 ack（4 字节）：期望收到的下一个序号
- 数据偏移、保留、标志位（SYN/ACK/FIN/RST/PSH/URG）
- 窗口大小（2 字节）：用于流量控制
- 校验和、紧急指针

## 三次握手（建立连接）

```
客户端                              服务器
  |                                   |
  | ---- SYN, seq=x ----------------> |  (1)
  |                                   |
  | <--- SYN+ACK, seq=y, ack=x+1 ---- |  (2)
  |                                   |
  | ---- ACK, ack=y+1 --------------> |  (3)
  |                                   |
```

1. 客户端发送 SYN 报文，seq=x，进入 SYN_SENT 状态。
2. 服务器收到后回复 SYN+ACK，seq=y, ack=x+1，进入 SYN_RCVD 状态。
3. 客户端发送 ACK，ack=y+1，双方进入 ESTABLISHED 状态。

**为什么三次而非两次？** 防止已失效的连接请求到达服务器后建立无效连接，浪费资源；同时确认双方收发能力都正常。两次握手无法防止历史连接的误建立。

## 四次挥手（释放连接）

```
客户端                              服务器
  |                                   |
  | ---- FIN, seq=u ----------------> |  (1) 主动关闭
  |                                   |
  | <--- ACK, ack=u+1 --------------- |  (2)
  |       (服务器继续发送剩余数据)    |
  |                                   |
  | <--- FIN, seq=v ----------------- |  (3)
  |                                   |
  | ---- ACK, ack=v+1 --------------> |  (4)
  |                                   |
  |     TIME_WAIT (2*MSL) 后关闭      |
```

1. 主动方发送 FIN，进入 FIN_WAIT_1。
2. 被动方回 ACK，进入 CLOSE_WAIT；主动方进入 FIN_WAIT_2。
3. 被动方数据发完后发送 FIN，进入 LAST_ACK。
4. 主动方回 ACK，进入 TIME_WAIT，等待 2MSL 后关闭。

**为什么 TIME_WAIT 要等 2MSL？** 一是确保最后一个 ACK 能到达对端（若丢失对端会重发 FIN）；二是让本次连接的所有报文都从网络中消失，防止干扰新连接。

**为什么挥手四次而握手三次？** 握手时服务器把 SYN 与 ACK 合并发送；挥手时服务器收到 FIN 后可能还有数据要发，所以 ACK 和 FIN 分开发送，故为四次。

## 可靠传输机制

### 序号与确认

TCP 对每个字节编号，接收方通过 ACK 确认已收到的数据。采用累积确认：ACK=n 表示序号 n 之前的所有数据均已收到。

### 重传

发送方启动定时器，超时未收到 ACK 则重传。超时时间 RTO 由 RTT 动态估算。

### 滑动窗口

滑动窗口实现流量控制。发送窗口 = min(接收方通告窗口, 拥塞窗口)。

- 发送方可在未收到确认前连续发送窗口内的数据，提高吞吐。
- 接收方通过 ACK 报文中的窗口字段通告可用缓冲区大小。
- 窗口为 0 时发送方停止发送（但仍发送零窗口探测报文，防止死锁）。

### 拥塞控制

四个经典算法：

1. **慢启动**：cwnd 从 1 开始，每收到一个 ACK 加倍增长，指数增长到 ssthresh。
2. **拥塞避免**：超过 ssthresh 后，每 RTT cwnd 加 1，线性增长。
3. **快重传**：收到 3 个重复 ACK 立即重传丢失报文，不等超时。
4. **快恢复**：ssthresh = cwnd/2，cwnd = ssthresh，进入拥塞避免（而非慢启动）。

超时事件触发时：ssthresh = cwnd/2，cwnd = 1，重新慢启动。

## UDP 特点

UDP 头部仅 8 字节：源端口、目的端口、长度、校验和。特点：

- 无连接，发送前无需建立连接；
- 不保证可靠交付，不重传；
- 头部开销小，传输效率高；
- 支持广播和多播；
- 没有拥塞控制，发送速率不受网络拥塞影响。

适用场景：实时音视频、DNS 查询、在线游戏、DHCP、TFTP。QUIC 在 UDP 之上自实现可靠传输，兼顾 UDP 的灵活与 TCP 的可靠性。

## 代码示例：UDP 客户端

```python
import socket

sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
sock.settimeout(2.0)
server = ('8.8.8.8', 53)

# 构造一个简单的 DNS 查询
query = b'\xaa\xbb\x01\x00\x00\x01\x00\x00\x00\x00\x00\x00' \
        b'\x06google\x03com\x00\x00\x01\x00\x01'
sock.sendto(query, server)
data, _ = sock.recvfrom(1024)
print('收到 %d 字节' % len(data))
sock.close()
```
