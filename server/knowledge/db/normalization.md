# 数据库范式

## 概述

数据库范式（Normal Form）是关系型数据库设计的规范，用于减少数据冗余、消除异常（插入/更新/删除异常）。范式从低到高逐级约束，常见的有 1NF、2NF、3NF、BCNF。

## 范式层级

### 1NF（第一范式）— 原子性

每个字段不可再分，是最基本的要求。

```
❌ 不满足 1NF：
| 学生 | 电话（座机+手机混合）  |
|------|----------------------|
| 张三 | 010-1234, 1380000    |

✅ 满足 1NF：
| 学生 | 座机       | 手机       |
|------|-----------|-----------|
| 张三 | 010-1234  | 1380000   |
```

### 2NF（第二范式）— 消除部分依赖

在 1NF 基础上，非主属性必须完全依赖于主键（不能只依赖主键的一部分）。

```
❌ 不满足 2NF（复合主键：学号+课程号）：
| 学号 | 课程号 | 姓名 | 课程名 | 成绩 |
|------|--------|------|--------|------|
| S1   | C1     | 张三 | 数据库 | 90   |

问题：姓名只依赖学号，课程名只依赖课程号 → 数据冗余

✅ 满足 2NF（拆分为三张表）：
学生表(学号, 姓名)
课程表(课程号, 课程名)
选课表(学号, 课程号, 成绩)
```

### 3NF（第三范式）— 消除传递依赖

在 2NF 基础上，非主属性不能传递依赖于主键。

```
❌ 不满足 3NF：
| 学号 | 姓名 | 院系编号 | 院系名称 |
|------|------|---------|---------|
| S1   | 张三 | D01     | 计算机系 |

问题：学号 → 院系编号 → 院系名称（传递依赖）→ 院系名称冗余

✅ 满足 3NF：
学生表(学号, 姓名, 院系编号)
院系表(院系编号, 院系名称)
```

### BCNF（BC 范式）— 消除主属性对候选键的依赖

在 3NF 基础上，每个决定因素都必须是候选键。

```
❌ 不满足 BCNF：
| 学生 | 课程     | 教师       |
|------|---------|-----------|
| 张三 | 数据库   | 王老师     |
| 李四 | 数据库   | 王老师     |

假设：一个课程只有一个教师，一个教师只教一门课程
函数依赖：课程 → 教师，教师 → 课程
候选键：(学生, 课程) 和 (学生, 教师)
问题：课程 → 教师，但课程不是候选键

✅ 满足 BCNF：
授课表(课程, 教师)
选课表(学生, 课程)
```

## 反范式设计

实际工程中不总是追求最高范式。为提升查询性能，有时故意增加冗余。

```sql
-- 订单表（反范式设计，直接存商品名称避免 JOIN）
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_name VARCHAR(50),      -- 冗余：本应通过 user_id JOIN users
    product_name VARCHAR(100),  -- 冗余：本应通过 product_id JOIN products
    quantity INT,
    price DECIMAL(10,2),        -- 冗余：本应从 products 表获取
    created_at TIMESTAMP
);
-- 优势：查询快（无需多表JOIN）
-- 代价：商品改名/改价时需同步更新所有订单
```

## 常见考点

1. **范式与反范式的权衡**：范式减少冗余但增加 JOIN，反范式提升查询性能但增加维护成本
2. **部分依赖 vs 传递依赖**：2NF 解决部分依赖（复合主键），3NF 解决传递依赖
3. **函数依赖**：X → Y 表示 X 决定 Y，如学号 → 姓名
4. **候选键 vs 主键**：候选键是最小超键，主键是选中的一个候选键
5. **实际应用**：通常满足 3NF 即可，数据仓库（OLAP）常使用反范式或星型模型
